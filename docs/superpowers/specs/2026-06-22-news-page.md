# 新闻/动态页 spec

**Date**: 2026-06-22
**Status**: approved（用户 2026-06-22 下午批准设计）
**Author**: Claude（基于 brainstorming + 5 个分段确认）

## Why

当前 wwwsite 缺少一个公司向用户传达信息的渠道：
- 所有页面顶 nav 都外链到 `https://blog.booming.one`（站外），SEO + 用户留存都差
- 现有 `admin-product-docs.html` 文档编辑能力（Quill + 草稿/发布 + 封面 + 分类）很成熟，**形态恰好是"内容管理"**，但绑定 product 维度
- 用户需要"公告/公司动态"场景的发布能力（产品发布、版本更新、公司新闻、安全公告），且不希望被产品维度束缚

**目标**：在站内新建"动态"模块，让 admin 能发布公告，前台用户通过 nav 的"动态"链接查看。

## What

### 1. 数据模型

**新表 `news`**（无 schema 变更影响现有表）：

```sql
CREATE TABLE news (
  id INT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  slug VARCHAR(255) NOT NULL UNIQUE,
  excerpt TEXT,
  content_html MEDIUMTEXT,
  cover_image VARCHAR(500),
  category VARCHAR(50),
  is_pinned BOOLEAN DEFAULT FALSE,
  status ENUM('draft', 'published') DEFAULT 'draft',
  view_count INT DEFAULT 0,
  published_at DATETIME NULL,
  sort_order INT DEFAULT 0,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_status_published (status, published_at DESC),
  INDEX idx_is_pinned (is_pinned)
);
```

**约束**：
- `slug` 全表 UNIQUE（防止 URL 冲突）
- `status` 枚举：draft（草稿）/ published（已发布）
- `published_at` 仅在 `publish` 时设置（首次发布），`unpublish` 不清空（保留发布时间戳）
- `is_pinned` 与 `sort_order` 双重排序：置顶优先 + 排序号次之 + published_at 兜底

### 2. API 层

#### Admin（需要登录）

| Method | Path | Body | 返回 | 用途 |
|--------|------|------|------|------|
| GET | `/api/admin/news` | — | `[{...}]` | 列表（草稿+已发布） |
| GET | `/api/admin/news/:id` | — | `{...}` | 单条 |
| POST | `/api/admin/news` | `{title, slug, excerpt, content_html, cover_image, category, is_pinned, sort_order}` | `{id}` | 创建（status=draft） |
| PUT | `/api/admin/news/:id` | 同 POST body | `{ok}` | 更新 |
| DELETE | `/api/admin/news/:id` | — | `{ok}` | 删除 |
| POST | `/api/admin/news/:id/publish` | — | `{ok}` | 发布（首次设 `published_at = NOW()`） |
| POST | `/api/admin/news/:id/unpublish` | — | `{ok}` | 撤稿（status=draft，**保留** published_at） |

#### 公开（无需登录）

| Method | Path | 返回 | 用途 |
|--------|------|------|------|
| GET | `/api/news` | `[{id, title, slug, excerpt, cover_image, category, is_pinned, published_at, view_count}]` | 列表（只 published） |
| GET | `/api/news?category=X` | `[{...}]` | 按 category 过滤（可选 query） |
| GET | `/api/news/:slug` | `{...全字段含 content_html}` | 单条详情 |
| POST | `/api/news/:id/view` | `{ok}` | view_count +1（**限流 30/min**） |

**列表响应**字段（公开）：不返 `content_html`，只返 `excerpt`（防大 payload）。详情才返 `content_html`。

### 3. admin 后台 UI

**新文件 `public/admin-news.html`**（约 320 行，镜像 `admin-product-docs.html` 但无 product select）：

**3 个区域**：
1. **工具栏**：分类过滤器（全部 / 产品更新 / 公司新闻 / 安全公告 / 自定义） + "+ 新建动态" 按钮 + "↗ 公开动态" 链接（target=_blank）
2. **列表**：每行显示 📌/无 + 标题 + slug + category + status badge（已发布/草稿）+ 编辑/发布/撤稿 按钮
3. **编辑器面板**：标题 / Slug（手动，限制 `^[a-z0-9-]+$`）/ 分类 select / 摘要 textarea / 封面图 URL input + 上传按钮 / ☐ 置顶 + 排序 input / 正文（Quill snow 主题 + image handler）/ [保存草稿] [发布] [删除] [取消] 4 按钮

**关键实现**：
- Quill 工具栏：`[{header:[2,3,false]}, bold/italic/underline/strike, {list:ordered/bullet}, blockquote/code-block, link/image, clean]`
- image handler：复用 `admin-product-docs.html` 的 5MB 限制 + `/api/upload-doc-image` 端点
- `escapeHtml(s)` 与 doc 同款防御
- 草稿/发布逻辑：保存草稿 → `PUT /api/admin/news/:id`；发布 → `PUT` 后再 `POST /publish`
- slug 唯一性：保存前 client 检查 + server 400 错误捕获（DB UNIQUE 兜底）

### 4. 公开页 UI

**新文件 `public/news.html`**（约 400 行，**单文件双模式** + URL 路由 `/news.html?slug=xxx`）：

#### 4.1 列表模式（默认）

```
┌─ 顶 nav（与 doc.html 一致，加"动态"链接） ─────────────┐
└──────────────────────────────────────────────────────────┘
┌─ 主区 ──────────────────────────────────────────────────┐
│  📢 博铭动态                                              │
│  产品更新 · 公司新闻 · 安全公告                              │
│  ┌─ 分类 tab：全部 12  | 产品更新 5  | 公司新闻 4 | ... ┐ │
│  ┌─ 📌 置顶卡片（大块）：封面 + 标题 + 摘要 + 阅读量 ────┐ │
│  ┌─ 普通卡片网格（2 列 / 1 列 mobile）：每张 6 篇 ──────┐ │
│  [< 上一页  1  2  3  下一页 >] 分页（每页 12）         │
└──────────────────────────────────────────────────────────┘
```

#### 4.2 详情模式（`?slug=xxx`）

```
┌─ nav ────────────────────────────────────────────────────┐
│  ← 返回动态列表                                            │
│  [封面图 full width]                                      │
│  分类 · 发布时间 · 👁 阅读量                                │
│  # 标题                                                    │
│  ┌─ 正文（max-width 800px 居中，DOMPurify 净化） ──────┐ │
│  无上一篇下一篇（YAGNI）                                  │
└──────────────────────────────────────────────────────────┘
```

**关键实现**：
- URL 路由：`location.search` 解析 `slug` query，无则列表模式，有则详情模式
- 数据 fetch：`/api/news?category=X`（按 tab）或 `/api/news/:slug`（详情）
- view_count：详情页 `DOMContentLoaded` 时 fire-and-forget `POST /api/news/:id/view`
- DOMPurify 净化（`/vendor/dompurify/3.0.6/purify.min.js`）渲染 `content_html`
- 封面图 fallback：无图时用 gradient 占位（category color）

## Out of Scope（YAGNI）

- ❌ 评论/点赞
- ❌ 全文搜索
- ❌ RSS / Atom feed
- ❌ 定时发布（手动 publish 即时生效）
- ❌ 多语言 i18n
- ❌ Newsletter 自动推送（已有 admin-newsletter.html 解耦）
- ❌ 草稿自动保存（手动点"保存草稿"）
- ❌ 版本历史/修订对比
- ❌ 上一篇/下一篇导航
- ❌ SEO meta 自动生成
- ❌ 批量操作（批量发布/删除）
- ❌ 复制动态
- ❌ admin 内嵌预览
- ❌ 自动 slug 拼音生成（admin 手动输入）
- ❌ 邮件订阅新动态通知
- ❌ 分类自定义管理页（自由输入 + 预设 tab 兜底）

## Risks

1. **slug 冲突**：并发创建同 slug → 400 + 友好提示（DB UNIQUE 约束兜底）—— 测试覆盖
2. **view_count 竞态**：用 `UPDATE ... SET view_count = view_count + 1` 原子 SQL（不读后写）
3. **Quill HTML 注入**：保存前不净化（admin trusted），**公开页详情渲染时** DOMPurify 净化
4. **封面图路径**：复用 `/uploads/docs/` 目录（不新建 `uploads/news/`）
5. **大量历史数据**：本次无历史（首次发布），无迁移负担
6. **侧栏 nav-news 位置**：放在 "产品管理" group **之外**（顶级），与"概览/支持"同层
7. **nav 链接需在多个页面同步**：约 7-8 个 HTML 文件需 grep `<a href="https://blog.booming.one"` 上下文批量加
8. **cache-bust 漏改**：25 个 admin HTML 都需 bump → 按 memory `feedback_admin_css_cache_bust.md` 强制
9. **公开 view 端点被刷**：限流 30/min（per memory `feedback_security_hardening.md` S12/S13 模式）
10. **view_count 大数溢出**：INT 上限 2.1B，普通公告不会触及；若需可改 BIGINT
11. **未登录访问 admin 端点**：复用现有 `requireAdmin` 中间件（per memory `feedback_security_hardening.md` 已有）
